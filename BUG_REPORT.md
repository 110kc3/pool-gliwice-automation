# BUG_REPORT.md

This report details potential bugs, weaknesses, and areas for improvement across the application based on a review of the source code, data pipeline, and CI/CD workflow. **No changes have been made to the codebase.**

---

## 🐞 1. Data Pipeline Analysis (master_aggregator.py)

The core data aggregation logic relies on sequentially reading multiple JSON files and transforming the resulting data structure.

### Potential Bugs & Weaknesses:

1.  **Error Handling Granularity (Critical):** The `aggregate_data` function catches any `Exception` when loading a file (`except Exception as e:`). This is too broad. If one file fails due to a malformed JSON or permissions issue, the error is printed, but the aggregation continues using only successfully loaded files, masking the root cause of the failure.
    *   **Suggestion**: Use more specific exceptions (e.g., `json.JSONDecodeError`) and potentially halt or flag the entire batch if a critical file like `mewa_data.json` fails to load, as the data contract implies all inputs must be present.
2.  **Data Consistency Assumption (High):** The logic assumes that if a JSON loads, it will contain entries that conform to the expected structure (e.g., `entry` is a dictionary with `"name"`, `"time"`, and `"days"`).
    *   **Weakness**: If a file contains entries missing the `"name"` key, the code attempts to backfill it (lines 60-64), but this is a manual patch, not a robust solution. If an entry is missing *all* keys, it might still be appended to `all_raw_data` and potentially crash later.
3.  **`defaultdict` Overwriting (Medium):** The use of `defaultdict(list)` correctly groups schedules by pool name. However, if two different input files (e.g., `delfin_data.json` and `olimpijczyk_data.json`) contain schedule data for the *same pool name* but for *different time slots*, they will be appended to the list for that pool name. This is likely intended, but if the requirement is for the *latest* time slot to overwrite previous ones, the current logic will compound them into a single list, which might be incorrect.

## 📊 2. Test Strategy & Coverage Gaps

No explicit testing framework commands (e.g., `pytest`, `npm test`) were found. The current workflow relies on manual runs or the limited checks in the CI.

*   **Gap**: There is no visible dedicated unit testing suite. The testing relies solely on the successful execution of the parsing/aggregation steps (`run: python master_aggregator.py`).
*   **Recommendation**: Unit tests should be added for `transform_data` in `master_aggregator.py` to test the list/dict manipulation in isolation from file I/O.

---

## ⚙️ 3. CI/CD Workflow Validation (`.github/workflows/weekly_update.yml`)

The workflow seems well-structured for a deployment process, but there are points of failure:

1.  **Dependency Management (Critical):** The dependencies (`pandas`, `numpy`, `tabula-py`) are installed using `sudo apt-get install` followed by `pip install`. This mix is brittle.
    *   **Bug Risk**: Using `sudo` in CI actions is discouraged. If the `apt-get` step modifies system libraries, it could affect subsequent steps unexpectedly. The dependencies should ideally be managed via a `requirements.txt` and installed cleanly using `pip install -r requirements.txt` within a contained Python virtual environment step, instead of relying on `sudo apt-get`.
2.  **Script Execution (Minor):** The scripts are called directly (`python parse_mewa.py`). This assumes Python is in the PATH and that all required libraries are installed globally/locally in the correct order.

---

## 🌐 4. Cross-Layer Boundary Check (Python $\leftrightarrow$ JavaScript)

This is the highest risk area for runtime bugs.

*   **Boundary**: The output of `master_aggregator.py` is written to `data.json`. This file is the contract that the frontend consumes via `index.html/script.js`.
*   **Potential Bug**: The structure generated by the backend (`result.append({...})` leads to an array of objects, where each object has a `schedule` key which is an array of objects containing `day`, `time`, and `availableLanes`). The frontend must rigidly adhere to this structure. If any backend change (e.g., renaming `availableLanes` to `lanes`) is made without updating the frontend parsing logic, the frontend will silently fail or render incorrect data.
*   **Recommendation**: The front-end must have robust checks for the shape of `data.json` on load, providing a clear fallback UI state if the data structure is unexpected.

---

### Summary of Actionable Bugs/Improvements:

1.  **Improve Exception Handling** in `master_aggregator.py`'s file loading loop.
2.  **Implement Unit Tests** for `transform_data`.
3.  **Refactor CI Dependencies** in `weekly_update.yml` to use virtual environments and rely on `requirements.txt` instead of mixed `sudo apt-get` and `pip` commands.
4.  **Implement Front-end Data Validation** in `script.js` (or associated JS logic) to handle deviations from the expected JSON schema.
